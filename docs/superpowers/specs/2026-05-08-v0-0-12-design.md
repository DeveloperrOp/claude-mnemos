# v0.0.12 — i18n alerts + Lost metric unification + placebo settings cleanup

> **Spec status:** approved 2026-05-08 by Ярик. Decomposes into a single
> implementation plan; subagent-driven execution.

## Goals

Three distinct fixes bundled into one release because they are independent
(no overlapping files), each small (0.5–2 h), and together address bugs the
user surfaced while live-testing v0.0.11.

1. **(A) Alerts i18n** — backend currently writes hardcoded English strings
   into `Alert.message`. Frontend renders the string verbatim, so users on
   a localized UI see English (e.g. `HOOK_SILENCE` warning popped up in
   Ярик's Ukrainian dashboard). Migrate to a key+params model so frontend
   localization handles the rendering.
2. **(B) Lost-sessions metric unification** — KPI bar on Overview shows
   `Lost: 1448` (= total uningested transcripts on disk) while the
   `Lost Sessions` page lists ~290 entries (= filtered by `mtime > 24h ago`
   and excluding `agent-*` sub-agent transcripts). Two numbers disagreeing
   ruins trust. Make the KPI use the same filter the user-facing page uses.
3. **(C) Remove placebo `ProjectSettings` groups** — the
   `frontend Settings → Project` page has a 12-section accordion. Five of
   those sections (`watchdog`, `ontology`, `lifecycle`, `prompts`,
   `telemetry`, `IngestOverrides`) write JSON to disk but are never read by
   any production code path (verified by grep across the entire repo).
   Remove them from the schema and UI so the page only shows toggles that
   actually do something.

Order of work: **B → A → C** (smallest/lowest-risk first).

## (A) Alerts i18n

### Current behaviour

`claude_mnemos/core/health_checks.py` builds alerts via:

```python
return _make(
    id_="hook_silence",
    detector="hook_silence",
    severity="warning",
    message=(
        f"{len(recent_jsonls)} recent Claude Code session(s) detected, but no "
        f"ingest job has succeeded in the last 6h. Check that hooks are installed."
    ),
    ...
)
```

`Alert.message` ends up in the JSON the dashboard fetches; the React card
renders it through plain `{alert.message}`. There is no place for the user's
locale to enter the picture. Even fully translated UI shells around the alert
keep the English body text intact.

### Target schema

Extend `Alert` with optional structured fields, keep `message` as an
English fallback so existing on-disk alerts (written by v0.0.11-) load
without migration:

```python
class Alert(BaseModel):
    id: str
    detector: str
    severity: Literal["info", "warning", "error"]
    message: str                                 # English fallback
    i18n_key: str | None = None                  # e.g. "alerts.hook_silence"
    i18n_params: dict[str, Any] | None = None    # e.g. {"count": 4}
    context: dict[str, Any] | None = None
    created_at: datetime
```

Detectors fill BOTH `message` and `i18n_key`+`i18n_params`. The frontend
prefers `i18n_key` when present:

```tsx
const text = alert.i18n_key
  ? t(alert.i18n_key, alert.i18n_params ?? {})
  : alert.message;
```

### Detector inventory

`grep -n '_make(' claude_mnemos/core/health_checks.py` plus the
`AlertsStore.add` call sites under `claude_mnemos/daemon/`. Confirmed
detectors in `health_checks.py`:

* `hook_silence` — params: `{count: int}`
* `hooks_not_installed` — params: none
* `disk_low` — params: `{vault: str, percent_free: float}`
* `daemon_offline` — params: none (info-level, used in onboarding gate)
* `watchdog_paused` — params: `{vault: str}`

Each detector picks its own key prefixed `alerts.<detector>`. Param names
match what the locale `t()` template expects.

### Locale keys

Add an `alerts` section to all three locale files (`uk.json`, `ru.json`,
`en.json`):

```json
"alerts": {
  "hook_silence": "{{count}} recent Claude Code session(s) detected, but no ingest job has succeeded in the last 6h. Check that hooks are installed.",
  "hooks_not_installed": "Mnemos hooks are not installed in ~/.claude/settings.json. Run mnemos init or fix from Settings → Hooks.",
  "disk_low": "Vault \"{{vault}}\" disk is {{percent_free}}% free.",
  "daemon_offline": "Daemon stopped responding.",
  "watchdog_paused": "Filesystem watcher paused for vault \"{{vault}}\"."
}
```

(Текст русской/украинской копии — отдельная задача автору переводов; в
implementation plan'е будут заглушки на английском с `TODO(i18n)`-комментами,
кроме примеров.)

### Frontend wiring

Single React component renders alert cards. Touch points:

* `frontend/src/types/alerts.ts` — extend the type to carry `i18n_key` and
  `i18n_params`
* `frontend/src/components/widgets/AlertCard.tsx` (or wherever the card
  lives — find it with grep) — read `i18n_key` first, fall back to `message`
* No backwards-compat shim on the type — make `i18n_key` optional and let
  TypeScript narrow at the call site.

### Tests

* `tests/state/test_alerts.py` (or equivalent) — assert detector emits
  both `message` and `i18n_key`/`i18n_params` and that they describe the
  same situation.
* `tests/core/test_health_checks.py` — update one detector test to assert
  the i18n payload alongside the legacy message.
* Frontend Vitest — render an `AlertCard` with `i18n_key` set, assert
  `t()` was called with the right key.

## (B) Lost metric unification

### Current behaviour

`claude_mnemos/daemon/routes/dashboard.py:snapshot` exposes
`kpi.lost_total`. Today it counts every `.jsonl` on disk that isn't in any
vault's manifest — that is, all uningested transcripts including:

* `agent-*` sub-agent fragments (filtered out from Lost Sessions UI per
  v0.0.8)
* sessions younger than 24 h (also filtered out per v0.0.8)

Meanwhile `claude_mnemos/daemon/routes/lost_sessions.py:_scan_all_vaults`
applies both filters and returns ~290 items for Ярик, while
`kpi.lost_total` says 1448. Same word, two numbers — bad UX.

### Target

`kpi.lost_total` MUST equal the count returned by `GET /lost-sessions`.
Concretely: dashboard's snapshot computes its lost_total by reusing the
same `_scan_all_vaults` helper (or its filtering logic), not the raw
uningested set.

### Implementation

* Move the filter helper out of `routes/lost_sessions.py` if it isn't
  already importable, or call the route function directly.
* `dashboard.py:snapshot` invokes the same scan + filter once and reads
  the resulting `len(filtered)`.
* No new field needed; `kpi.lost_total` keeps the same name with a
  more useful value.
* If keeping the raw uningested count is desirable for a debug page, expose
  it under a different name (`kpi.uningested_total`). Out of scope for
  v0.0.12 — only do if zero-cost.

### Tests

* `tests/daemon/test_routes_dashboard.py` — assert `kpi.lost_total` matches
  the size of `GET /api/lost-sessions` body for the same vault state. The
  two endpoints must agree by construction.

## (C) Remove placebo `ProjectSettings` groups

### Current behaviour

`claude_mnemos/state/settings.py:ProjectSettings` has these subgroups:

* `auto_ingest` — read by hooks/auto-dump as of v0.0.10/11 ✅
* `lint` — read by lint runner ✅
* `snapshots` — read by daily-snapshot scheduler ✅
* `watchdog` — **never read** ❌
* `ontology` — **never read** ❌
* `lifecycle` — **never read** ❌
* `prompts` — **never read** ❌
* `telemetry` — **never read** ❌
* `ingest` (`IngestOverrides`) — **never read** ❌

Frontend renders all 9 groups in a Project Settings accordion. PATCH
endpoints accept all 9. JSON files on disk persist all 9. None of the
five "❌" groups influence any code path — purely decorative.

### Target

Drop the five unused groups from:

* `ProjectSettings` model (`claude_mnemos/state/settings.py`)
* the SettingsStore round-trip (no special handling needed beyond schema)
* the dashboard's Settings page accordion
* locale strings naming those sections

The user's existing `~/.claude-mnemos/settings/<project>.json` files may
contain these fields — `model_config = ConfigDict(extra="ignore")` already
swallows them silently. Add a one-time test to confirm.

### Bonus cleanup (same release, same theme)

* `claude_mnemos/state/install_state.py:InstallState.first_session_celebrated_for`
  + the `mark_celebrated()` method — never called in production. Drop.
* `claude_mnemos/daemon/config.py:migrate_legacy_dotmnemos()` — α-period
  migration that no caller invokes any more. Drop the function and its
  test file.

### Tests

* `tests/state/test_settings.py` — drop tests asserting `watchdog`,
  `ontology`, `lifecycle`, `prompts`, `telemetry`, `ingest` defaults.
* Add a forward-compat test: load a v0.0.11-shape JSON with `watchdog`
  field present, verify it loads cleanly with `extra="ignore"` on the
  affected groups (or at the top level).
* Drop `tests/state/test_install_state.py::test_mark_celebrated*` if
  present.
* Drop `tests/daemon/test_pid_file_migration.py` (covered the dropped
  migrate function).
